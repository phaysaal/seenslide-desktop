Part I — Non-GUI automated testsI.1 Methodology

LayerToolRuns againstGate
Unit / integrationpytest (tests/, 65 tests)in-process, temp dirs, mocks CI on every push/PR and required by all three release builds
HTTP securitytests_gui_agent/w6_auth_security.pyproduction seenslide.com manual; throwaway accounts, self-purging
DB atomicitytesting_scripts/test_credit_transactions.pyproduction PostgreSQL manual; throwaway rows, self-purging
  • Test the real thing when the risk is integration. Auth-token semantics and transaction atomicity are properties of the deployed system (FastAPI + asyncpg + Postgres together), so those tests run against production with disposable data — mocks would encode the same wrong assumptions as the code.
  • Throwaway data must be self-evidently throwaway. Test accounts use @harness.invalid (RFC 2606 reserved TLD — can never receive mail or collide with a person); test rows use tx-test-<uuid> ids. Purges are name-gated inside the SQL itself, never lists of ids collected elsewhere.
  • Regression tests are written at fix time. Every unit-testable bug fixed during this effort got a pytest module the same day.

I.3 HTTP-level auth security

The web viewer authenticates by magic link — the email is only an identifier, no verification mail matters, the token is the credential. The security properties live at the HTTP layer, so they are tested there: the freshly-created token is read from the magic_links table (exactly what an inbox would have received) to complete login without any mailbox.

Checks (production, two @harness.invalid accounts, auto-purged): login issues a working session · the token is single-use (replay rejected) · the accounts are distinct user_ids · logout invalidates the session (401 afterwards).

The "wrong-PIN lockout" (MAX_FAILED_ATTEMPTS=10, 15-min lock) is a desktop/phone identity feature — the web path has no PIN, so it is not a web test.

I.4 DB-level credit atomicity

Exercises the real CreditManager through DatabaseManager.transaction() against the production database with a throwaway user: grant (+50, one logged transaction), deduct (−20, second transaction), insufficient-funds refusal (balance untouched), and the decisive rollback test — inside a transaction the uncommitted balance is visible (+500); a forced error rolls it back exactly to the pre-transaction value.

Part II — LLM-driven GUI testingII.1 The idea

Conventional GUI automation binds tests to selectors and accessibility ids — brittle, and impossible for surfaces you don't control (native file dialogs, browser chrome, two different browsers at once). This harness splits the problem:

  • Control flow is deterministic: scenarios are prewritten YAML step lists. No agent improvisation — a run either follows the script or fails.
  • Perception is an LLM: a vision model receives a screenshot and a natural-language description ("the red 'End Presentation' button") and returns coordinates (locate), a boolean judgment (judge), or a value read off the pixels (read — e.g. the 8-character join code shown in a dialog).
  • Truth is never only visual: every scenario ends in hard oracles — SQLite queries, public HTTP endpoints, perceptual image hashing against the source PDF, audio waveform peak analysis — so a plausible-looking screen can't fake a pass.

II.2 Architecture (tests_gui_agent/)

scenario.yaml ──▶ runner.py ──▶ locator.py ──▶ claude -p (opus-4.8, vision)
                     │               └─ coords cache (--replay = 0 AI calls)
                     ├─▶ actions.py     xdotool click/type/key/drag · mss screenshots
                     │                  browser ACTORS (window-cropped shots, offset clicks)
                     │                  sandboxed App lifecycle
                     ├─▶ presenter.py   fitz PDF → fullscreen Qt window ("the human presenting")
                     ├─▶ audio.py       piper-TTS + pw-loopback virtual microphone
                     ├─▶ relay.py       local HTTP relay → cloud (kill = network outage)
                     └─▶ report_html.py film-strip HTML report (bbox + crosshair per step)
  • Locator. Model claude-opus-4-8 (localizing labeled buttons doesn't need the top tier; ~2× faster). Backend: the claude -p CLI headlessly, or the Anthropic SDK when ANTHROPIC_API_KEY is set. ~5–15 s per call. Coordinates cached by (mode, screen size, description); --replay reruns with zero locate calls. read results are never cached — values change per run.
  • Input. xdotool synthesizes OS-level events — indistinguishable from a human, so the app's input-monitor capture trigger fires naturally. Primary monitor via xrandr; offset-corrected clicks; verified on 1- and 2-monitor setups.
  • Sandboxing. Fresh temp $HOME; PYTHON_KEYRING_BACKEND=null (the real Secret Service once popped the developer's password-manager dialog mid-run); a random pre-seeded device id for cloud runs — the machine-id fallback would silently resurrect the developer's real anonymous account; SEENSLIDE_TEST_ON_TOP=1 (Mutter ignores external raise requests). The sandbox config is the real project config with storage redirected — a minimal config once hid bug #1.
  • Presenter. Renders PDF pages fullscreen on a chosen monitor with a schedule (hold-first / interval / start-page for Next-Talk segments). Must bind the Qt window to the target QScreen and use showFullScreen() — Mutter mis-places frameless setGeometry windows on multi-monitor.
  • Synthetic voice. piper-TTS speaks one line per PDF page, silence-padded to the exact schedule. A pw-loopback pair (sink → device-class source "SS-Test-Mic") becomes the system default microphone and is always restored. Only working recipe on PipeWire: pw-cli create-node silently creates nothing, and wireplumber refuses stream-class nodes as default. Audio output is verified by recording the default sink's monitor (stream.capture.sink=true) and asserting waveform peaks — including a max_peak mode that proves silence.
  • Network fault injection. The sandbox app reaches the cloud through a local HTTP relay; killing it is a clean outage (instant connection-refused). Pitfall: the relay must strip Content-Encoding (requests pre-decodes upstream bodies).
  • Two-browser actors. actors: {A: edge, B: firefox} binds names to real browser windows by title + browser marker (largest match wins, self-verified by an AI judge). Targeted steps crop the screenshot to that window's rect (clamped — Mutter parks windows a few px offscreen) and offset clicks to the window origin: two viewers of the same page can't be confused, and nothing outside the window is ever clicked (the developer's terminal shared a monitor with actor B throughout).
  • Step DSL. launch · kill · wait · click (retries/optional) · type ($FIXTURES/$VAR) · key · assert_screen · read_text → $VAR · draw · actors · present_pdf (narrate/start_page) · wait_presenter · virtual_mic · play_audio · relay · show_cloud_code → $CLOUD_URL
  • Oracles. verify_db (counts, per-talk 1..N) · verify_slides_match_pdf (best-match dhash vs source pages + presentation-order check) · verify_cloud (public endpoint: what the audience sees; slides + viewers) · verify_talks (conference split; first visible slide of each talk equals its title page) · verify_voice (app log + WAV peak) · verify_hidden (gate armed; cloud == kept exactly) · verify_outbox · verify_audio_out (sink peak / silence) · verify_log (regex proof).
  • Artifacts per run: numbered screenshots, film-strip HTML report, sandbox app log, sandbox DB (saved via the sqlite3 backup API — a plain copy of a WAL-mode DB is empty), presenter and relay logs, narration + recorded WAVs, report.json.

II.3 Methodology rules learned the hard way

  • State-tolerant steps. Runs fail mid-way; the next run inherits the leftovers. Steps that open/join/play must be optional: or leave-then-redo — a voice call surviving from a previous run holds a dead mic node and transmits silence.
  • Don't interact during continuous playback. Locator think-time (5–15 s) is real time: a marker-walk during playback drifted past markers nondeterministically. Pause first, then walk.
  • Distinguish product bugs from oracle bugs. Several "failures" were wrong assertions (a deck whose page 4 is a title slide; a voice bar that has no speaking indicator). Discipline: read the artifacts before touching product code.
  • Locator flakiness exists. ~1 refusal to find a plainly visible element per hundreds of calls; a plain re-run passed. Policy: re-run once before treating a locate-miss as a regression.
  • Cleanup discipline. Purge only data whose harness origin is provable in the delete statement itself — never ids collected from logs. This rule exists because a session collected from a log line was wrongly deleted and had to be reconstructed from surviving volume files. Also: the server caches sessions in memory — DB-level deletes keep serving until a redeploy.
  • The presenter covers a monitor. The web actor sharing it joins live talks early and is verified after the deck ends; the unobstructed actor does the mid-talk checks.
  • Autoplay policies are features. Live audio and replay need a user gesture; scenarios click "Listen Live"/play rather than reporting silence as a bug.

II.6 Cost & performance profile

  • 7–25 AI calls per scenario; ~5–15 s per call; scenarios run 1–5 minutes wall-clock.
  • --replay reruns from the coordinate cache with zero locate calls.
  • The locator runs on a Claude subscription via claude -p; an ANTHROPIC_API_KEY switches it to the metered API with no other change.

III.2 What the effectiveness demonstrates

  • The bugs were integration bugs. Not one of the ten was findable by the unit suite alone: config plumbing (#1), event-wiring branches (#2), theme × widget interaction (#3), compositor timing (#4), cross-repo counter bookkeeping (#5), memory-vs-DB divergence (#6), and dead wiring that type-checks fine (#7–#9). The harness found them because it exercises the same seams a user does — and its hard oracles caught what looked fine on screen.
  • Bug cascades are real. #8 → #9 → #10 is a chain where each fix exposed the next; only end-to-end verification after each fix caught the follow-ons.
  • Vision-LLM perception is production-grade for this use. Across ~30 scenario runs: zero misclicks that damaged state; failures were overwhelmingly correct rejections. One flake pattern is handled by a re-run policy.
  • One step vocabulary spans surfaces nothing else covers uniformly: a PyQt app, a native GTK file dialog, browser permission prompts, two different browsers side-by-side, WebRTC audio, and a live production backend.

III.3 Known limits

  • Linux/X11 only today (xdotool + mss). Windows/macOS and the Wayland capture path remain untested by the harness.
  • The harness shares the screen: runs need the monitor hands-off; a human moving windows mid-run is the main environmental hazard.
  • Locator latency makes real-time interaction tests nondeterministic — design scenarios to pause first.
  • Judgments are natural language: an imprecise assert can pass for the wrong reason. The mitigation is the hard-oracle layer every scenario ends with.
